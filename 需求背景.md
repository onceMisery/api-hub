# 事件标识升级说明

## 目标

这次升级的目标是：在不要求现有事件类和发布代码做任何修改的前提下，引入一个可选的稳定事件标识，用于提升历史事件的长期回放兼容性。

## 变更内容

事件存储表现在会持久化三个身份相关字段：

- `event_class`：Java 类名，作为兼容兜底字段保留
- `event_name`：用于稳定重放的逻辑事件名
- `event_version`：逻辑事件版本号

如果事件类没有声明 `@EventIdentity`，框架会自动回退为：

- `event_name = event.getClass().getName()`
- `event_version = 1`
- `event_class = event.getClass().getName()`

这意味着现有使用者不需要修改事件类，也不需要调整发布代码。

## 可选注解

只有当某个事件需要脱离 Java 类名、拥有稳定的逻辑身份时，才需要添加 `@EventIdentity`。

```java
import com.ec.saas.infra.event.IEvent;
import com.ec.saas.infra.event.annotation.EventIdentity;

@EventIdentity(name = "user.created", version = 1)
public class UserCreatedEvent implements IEvent {
}
```

当事件类声明了这个注解后：

- 新事件会持久化 `event_name = "user.created"`
- 新事件会持久化 `event_version = 1`
- `event_class` 仍然会继续保存，作为兜底字段
- 重试回放时会优先使用 `event_name + event_version`，找不到时再回退到 `event_class`

## 扫描规则

为了尽量不影响现有使用者代码，`@EventIdentity` 不需要额外配置。

当前版本会自动扫描：

- 应用基础包
- 已注册 Spring Bean 所在模块包

这样即使事件类定义在依赖模块中，只要该模块已经通过 Spring Bean 接入当前应用，稳定事件标识也能在回放阶段被解析到。

## 升级顺序

如果当前表结构里还没有 `event_class` 字段：

1. 先消费完或清理所有待处理事件。
2. 停止应用。
3. 执行 [event_driver_upgrade.sql](/C:/user-data/code/yks/saas-infrastructure/saas-infrastructure-event/event_driver_upgrade.sql)。
4. 执行 [event_driver_identity_upgrade.sql](/C:/user-data/code/yks/saas-infrastructure/saas-infrastructure-event/event_driver_identity_upgrade.sql)。
5. 部署新版本应用。

如果当前表结构里已经有 `event_class` 字段：

1. 先消费完或清理所有待处理事件。
2. 停止应用。
3. 执行 [event_driver_identity_upgrade.sql](/C:/user-data/code/yks/saas-infrastructure/saas-infrastructure-event/event_driver_identity_upgrade.sql)。
4. 部署新版本应用。

## 兼容性说明

- 现有事件类即使不加 `@EventIdentity` 也仍然可以继续工作。
- 现有事件发布调用方式不需要调整。
- `routeKey` 仍然只是路由字段，不应作为稳定事件标识使用。
- `COMPOSITE` 事件当前仍然使用单条聚合状态记录。某个分支成功、另一个分支失败时，成功分支在后续重试中可能被再次投递，因此消费方必须保证幂等。

## 建议用法

- 如果历史事件类短期内不会改名，可以保持旧事件不变。
- 对于新事件或未来可能频繁演进的集成事件，建议增加 `@EventIdentity`。
- 只有在事件载荷发生不兼容变化时，才递增 `version`。
